Apache Tomcat is a free, open-source web server and servlet container developed by the Apache Software Foundation, used to host Java-based web applications and implements Java Servlet and JavaServer Pages (JSP) specifications.
Here's a more detailed explanation:

What it is:
Tomcat is a core component in the Java web development ecosystem, acting as a bridge between web servers and Java-based applications.
Functionality:
It receives HTTP requests, processes them using Java servlets and JSP, and sends back the response to the client.
Key features:
Open-source and free: Developed and maintained by the Apache Software Foundation.
Servlet container: It implements the Java Servlet API, allowing developers to create dynamic web content using Java.
JSP engine: It also supports JavaServer Pages (JSP), a technology for creating web pages with dynamic content.
Lightweight and efficient: Tomcat is designed to be a lightweight and efficient solution for hosting Java web applications.
Scalable and robust: It is known for its scalability and robustness, making it suitable for various web application deployments.
Widely used: It's a popular choice for hosting Java web applications, with a large and active community supporting it.

Beans:- In Spring, the objects that form the backbone of your application and that are managed by the Spring IoC container are called beans.
        A bean is an object that is instantiated, assembled, and managed by a Spring IoC container. Otherwise, a bean is simply one of many objects in your application.
        In Spring Boot, "beans" are simply Java objects managed by the Spring IoC (Inversion of Control) container, which handles their instantiation, configuration, and lifecycle.
        They are the fundamental building blocks of a Spring application and represent various components like controllers, services, and repositories.

ORM:- is a technique used to map java objects to databases tables. it allows developers to work with databases using object-oriented programming concepts, making it easier to interact with relational databases
JPA:- is a way to achieve ORM, includes interfaces and annotations that you use in java classes, requires a persistence provider(ORM tools) for implementation.
      Java Persistence API, is a specification for managing relational data in Java applications. It provides a standard interface for mapping Java objects to database tables, simplifying database interactions. JPA is part of Jakarta EE (formerly Java EE) and is not a tool or framework itself, but rather defines concepts for persistence.

      To use JPA, you need a persistence provider. A persistence provider is a specific implementation of the JPA persistence providers include Hibernate, EclipseLink, and OpenJPA. these providers implement the JPA interfaces and provide the underlying functionality to interact with databases.

Spring JPA :- Spring Data JPA is a build on top of the JPA(java persistence api) specification, but it is not a JPA implementation itself. instead, it simplifies working with JPA by providing higher-level abstractions and utilities.
              However, to use spring Data JPA effectively, you still need a JPA implementation, such as Hibernate, EclipseLink, or another JPA-compliant provider, to handle the actual database interactions.

///----But in case of MongoDB which is nosql db, you don't need JPA
In case of MongoDB, you don't have a traditional JPA persistence provider. MongoDB is a NoSQL database, and Spring Data MongoDB serves as tha "persistence provider" for MongoDB. it provides the necessary abstractions and implementations to work with mongodb in a spring application.

//**************( Spring Security )******************************************************************
at start, spring-boot-starter-security framework automatically give password in console


methods:-
  (i) http.authorizeRequest() :- this tells spring security to start authorizing the requests.
  (ii) .requestMatchers("/hello").permitAll :- this part specifies that HTTP requests matching the path /hello should be permitted(allowed) for all users, whether they are authenticated or not.
  (iii) .anyRequest().authenticated() :- this is a more general matcher that specifies any request ( not already matched by previous matchers) should be authenticated,  meaning users have to provide valid credentials to access these endpoints.
  (iv) .and() :- this is a method to join several configurations. it helps to continue the configuration from the root(HTTP Security)
  (v). formLogin() :- this enables form-based authentication. By default, it will provide a form for the user to enter their username and password. if the user is not authenticated and they try to access a secured endpoints, they'll be redirected to the default login form.
